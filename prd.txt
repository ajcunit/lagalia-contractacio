PRD ACTUALIZADO - Sistema de Gestión de Contratos Públicos
1. Visión General
Sistema de gestión para visualizar y administrar información de contratos públicos provenientes de la API de Transparència Catalunya, permitiendo a departamentos y sus empleados gestionar contratos asociados mediante sincronización local.
2. Objetivos del Producto

Centralizar la información de contratos públicos en una base de datos local
Permitir gestión departamental de contratos y personal
Detectar automáticamente duplicados para validación manual
Sincronizar datos periódicamente sin consultar la API en cada operación

3. Usuarios Objetivo

Responsables de Departamento: Gestionan empleados y validan contratos
Empleados: Consultan contratos asociados a su departamento
Administradores del Sistema: Dan de alta departamentos y gestionan sincronizaciones

4. Funcionalidades Principales
4.1 Gestión de Departamentos
Alta de Departamentos

Crear departamentos con información básica (nombre, código, descripción)

Desactivar/activar departamentos

4.2 Sincronización de Datos
Proceso de Sincronización

Consulta periódica al endpoint de la API
Comparación con datos locales existentes (usando hash del registro completo)
Detección de cambios (INSERT, UPDATE, DELETE)
Actualización incremental de la base de datos

Detección de Duplicados

Identificar contratos con mismo codi_expedient Y mismo estat_actual
Marcar automáticamente como "Pendiente de Aprobación"
Notificar a responsables del departamento

Configuración

Rollback en caso de errores críticos
Filtrado por codi_ine10 si se gestionan múltiples entidades

Campos de Control de Sincronización

data_actualitzacio: para detectar cambios desde última sincronización
data_publicacio: para ordenar cronológicamente
Hash MD5 del registro completo para detectar modificaciones

CREATE TABLE sincronizaciones (
    id SERIAL PRIMARY KEY,
    fecha_hora_inicio TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    fecha_hora_fin TIMESTAMP,
    registros_nuevos INTEGER DEFAULT 0,
    registros_actualizados INTEGER DEFAULT 0,
    registros_sin_cambios INTEGER DEFAULT 0,
    estado VARCHAR(50) CHECK (estado IN ('en_proceso', 'exitosa', 'fallida', 'parcial')),
    log_errores TEXT,
    url_endpoint TEXT,
    total_registros_api INTEGER,
    CONSTRAINT chk_fechas CHECK (fecha_hora_fin IS NULL OR fecha_hora_fin >= fecha_hora_inicio)

-- Duplicados detectados
CREATE TABLE duplicados (
    id SERIAL PRIMARY KEY,
    contrato_id_1 INTEGER REFERENCES contratos(id) ON DELETE CASCADE,
    contrato_id_2 INTEGER REFERENCES contratos(id) ON DELETE CASCADE,
    campo_duplicado VARCHAR(100), -- 'codi_expedient'
    valor_duplicado VARCHAR(500),
    fecha_deteccion TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    estado_validacion VARCHAR(50) CHECK (estado_validacion IN ('pendiente', 'aprobado', 'rechazado', 'fusionado')),
    usuario_validador_id INTEGER REFERENCES empleados(id),
    fecha_validacion TIMESTAMP,
    accion_tomada VARCHAR(50), -- 'aprobar_1', 'aprobar_2', 'fusionar', 'rechazar_ambos'
    observaciones TEXT,
    CONSTRAINT unique_duplicado UNIQUE (contrato_id_1, contrato_id_2)
);

-- Historial de cambios (auditoría)
CREATE TABLE historial_contratos (
    id SERIAL PRIMARY KEY,
    contrato_id INTEGER REFERENCES contratos(id) ON DELETE CASCADE,
    campo_modificado VARCHAR(100),
    valor_anterior TEXT,
    valor_nuevo TEXT,
    usuario_id INTEGER REFERENCES empleados(id),
    fecha_modificacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    tipo_cambio VARCHAR(50) CHECK (tipo_cambio IN ('sincronizacion', 'manual', 'validacion'))
);

-- Reglas de asociación automática
CREATE TABLE reglas_asociacion (
    id SERIAL PRIMARY KEY,
    departamento_id INTEGER REFERENCES departamentos(id),
    tipo_regla VARCHAR(50) CHECK (tipo_regla IN ('departamento', 'organismo', 'palabra_clave', 'cpv', 'importe')),
    campo_origen VARCHAR(100), -- 'departament_adjudicador', 'cpv_principal_codi', etc.
    valor_buscar TEXT,
    operador VARCHAR(20) CHECK (operador IN ('igual', 'contiene', 'comienza_con', 'mayor_que', 'menor_que')),
    prioridad INTEGER DEFAULT 0,
    activa BOOLEAN DEFAULT true,
    fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

## 6. Lógica de Sincronización Detallada

### 6.1 Algoritmo de Sincronización
```
1. Iniciar registro de sincronización
   - INSERT en tabla sincronizaciones (estado: 'en_proceso')

2. Consultar API
   - GET https://analisi.transparenciacatalunya.cat/resource/ybgg-dgi6.json?codi_ine10=4305160009
   - (Agregar parámetros de paginación si dataset es muy grande)

3. Para cada registro de la API:
   a. Calcular hash_contenido = MD5(JSON completo del registro)
   
   b. Buscar en BD por codi_expedient
   
   c. Si NO existe:
      - INSERT nuevo contrato
      - Aplicar reglas de asociación automática
      - Incrementar contador registros_nuevos
   
   d. Si existe:
      - Comparar hash_contenido
      - Si hash difiere:
        * UPDATE todos los campos
        * Guardar cambios en historial_contratos
        * Actualizar fecha_ultima_sincronizacion
        * Incrementar contador registros_actualizados
      - Si hash igual:
        * Incrementar contador registros_sin_cambios
   
   e. Detectar duplicados:
      - Buscar otros contratos con mismo codi_expedient Y mismo estat_actual
      - Si encuentra duplicado:
        * Marcar ambos con estado_interno = 'pendiente_aprobacion'
        * INSERT en tabla duplicados
        * Enviar notificación a responsables

4. Finalizar sincronización
   - UPDATE registro sincronizaciones (estado: 'exitosa', fecha_hora_fin)
   - Generar resumen de cambios
```

### 6.2 Campos Críticos para Detección de Duplicados

**Criterio Principal**: `codi_expedient` + `estat_actual`

**Campos a Comparar en Panel de Validación**:
- Importes: `preu_licitar` vs `preu_adjudicar` vs `import_adjudicacio_amb_iva`
- Adjudicatario: `adjudicatari_nom`, `adjudicatari_nif`
- Fechas: todas las `data_*` para ver evolución temporal
- Organismo: `organisme_adjudicador`, `departament_adjudicador`

## 7. Flujos Actualizados

### 7.1 Flujo de Sincronización

1. Sistema ejecuta sincronización (manual desde UI o programada)
2. Consulta endpoint con filtro `codi_ine10` si aplica
3. Para cada registro recibido:
   - Calcula MD5 del JSON completo
   - Compara con hash almacenado
   - Si difiere o no existe: actualiza/inserta
4. Detecta duplicados por `codi_expedient` + `estat_actual`
5. Aplica reglas de asociación automática a contratos nuevos
6. Genera log detallado con estadísticas
7. Notifica a responsables si hay duplicados pendientes

### 7.2 Flujo de Validación de Duplicados

1. Responsable accede a panel "Contratos Pendientes de Aprobación"
2. Sistema muestra pares de contratos duplicados con comparativa:
```
   Contrato A                    | Contrato B
   -----------------------------|------------------------------
   Expediente: EXP123           | Expediente: EXP123
   Estado: Adjudicado           | Estado: Adjudicado
   Precio Adjudicación: 50.000€ | Precio Adjudicación: 52.000€
   Fecha Adjudicación: 01/2025  | Fecha Adjudicación: 01/2025
   Adjudicatario: Empresa A     | Adjudicatario: Empresa A
```
3. Responsable analiza diferencias (probablemente uno es actualización del otro)
4. Decide acción:
   - **Aprobar más reciente**: marca el contrato con `data_actualitzacio` más reciente como válido
   - **Rechazar desactualizado**: marca el otro como rechazado
   - **Fusionar**: consolida información (toma campos más actualizados)
5. Sistema registra decisión en tabla `duplicados` y actualiza `estado_interno`

### 7.3 Flujo de Asociación Automática

1. Nuevo contrato insertado tras sincronización
2. Sistema evalúa reglas activas ordenadas por prioridad
3. Para cada regla:
```
   Si regla.tipo = 'departamento':
     Si contrato.departament_adjudicador CONTIENE regla.valor_buscar:
       SET contrato.departamento_id = regla.departamento_id
   
   Si regla.tipo = 'cpv':
     Si contrato.cpv_principal_codi = regla.valor_buscar:
       SET contrato.departamento_id = regla.departamento_id
   
   Si regla.tipo = 'palabra_clave':
     Si contrato.objecte_contracte CONTIENE regla.valor_buscar:
       SET contrato.departamento_id = regla.departamento_id
```
4. Si ninguna regla aplica, queda sin asignar para revisión manual

9. Reglas de Negocio Adicionales
9.1 Detección Inteligente de Duplicados
Además del criterio básico (codi_expedient + estat_actual), considerar:
Casos especiales:

Si dos contratos tienen mismo expediente y estado, pero data_actualitzacio difiere por menos de 24h → probablemente es actualización, priorizar más reciente
Si importes difieren en más del 10% → marcar como crítico para revisión
Si adjudicatarios son diferentes → marcar como crítico (posible error)

9.2 Estados del Contrato en el Ciclo de Vida
Según los datos de la API, los estados típicos son:

Anuncio previo
Licitación
Adjudicación
Formalización
(otros según estat_actual)

Lógica: Un mismo expediente puede tener múltiples registros en diferentes estados (evolución del contrato). Solo se considera duplicado si estado Y expediente coinciden.

10. Consideraciones Técnicas Adicionales
10.1 Rendimiento
Optimizaciones necesarias:

Índices en campos de búsqueda frecuente
Paginación: 50 registros por página por defecto
Búsqueda con debounce (300ms)
Cache de filtros aplicados
Para datasets grandes (>10K), considerar sincronización incremental usando data_actualitzacio como filtro

10.2 Manejo de Errores
Errores posibles:

API no disponible → reintentar 3 veces con backoff exponencial
Timeout → abortar sincronización, marcar como fallida
Datos corruptos/malformados → log error, continuar con siguiente registro
Violación de constraint → log error, notificar admin

10.3 Logs y Auditoría
Eventos a registrar:

Todas las sincronizaciones (exitosas y fallidas)
Validaciones de duplicados (quién, cuándo, qué acción)
Reasignaciones de contratos entre departamentos
Cambios en reglas de asociación
Accesos a contratos (opcional, para compliance)

Anexo: Ejemplos de Queries SQL Útiles
sql-- Contratos con duplicados pendientes de validación
SELECT c.codi_expedient, c.estat_actual, COUNT(*) as total_duplicados
FROM contratos c
WHERE c.estado_interno = 'pendiente_aprobacion'
GROUP BY c.codi_expedient, c.estat_actual
HAVING COUNT(*) > 1;

-- Contratos sin asignar a departamento
SELECT codi_expedient, objecte_contracte, preu_adjudicar
FROM contratos
WHERE departamento_id IS NULL
AND estado_interno = 'normal'
ORDER BY preu_adjudicar DESC;

-- Resumen de última sincronización
SELECT 
    fecha_hora_inicio,
    registros_nuevos,
    registros_actualizados,
    registros_sin_cambios,
    (registros_nuevos + registros_actualizados + registros_sin_cambios) as total_procesados
FROM sincronizaciones
ORDER BY fecha_hora_inicio DESC
LIMIT 1;

-- Top 10 adjudicatarios por volumen
SELECT 
    adjudicatari_nom,
    COUNT(*) as total_contratos,
    SUM(import_adjudicacio_amb_iva) as volumen_total
FROM contratos
WHERE adjudicatari_nom IS NOT NULL
GROUP BY adjudicatari_nom
ORDER BY volumen_total DESC
LIMIT 10;
